## 실험① BM25 vs 임베딩 (Week 6)

- **가설 예시**: "키워드형 질문은 BM25가, 패러프레이즈형은 임베딩이 hit@3에서 우세할 것"
- **바꾸는 것**: 검색 방식만. **고정**: chunking(500/100), k, 골든셋

| 검색 방식 | hit@3 | hit@5 | 실패 문항 id |
|---|---|---|---|
| (선택) v0 베이스라인 — 단어 겹침 개수 방식 | | | |
| BM25 (공백 토큰) | | | |
| 임베딩 (ko-sroberta) | | | |

> 💡 BM25 이전에 만들어둔 단순 검색기(질문 단어가 chunk에 있으면 +1점)가 있다면 버리지 말고 **베이스라인 행**으로 넣으세요. "겹침 개수 → BM25로 hit@3가 X→Y 개선"은 면접에서 가장 말하기 좋은 스토리입니다 (IDF와 길이 보정이 왜 필요한지를 숫자로 증명한 것이므로).

- **추가 분석 (핵심!)**: 실패 문항을 유형별로 나눠 "어느 유형에서 갈렸나" 표 하나 더
- 관문 질문 연결: *"왜 어떤 질문은 BM25가, 어떤 질문은 embedding이 잘 맞았는가?"*

## 실험③ chunk size 3종 (Week 7)

- **가설 예시**: "작은 chunk는 정밀하지만 문맥이 잘려, 중간 크기가 최적일 것"
- **바꾸는 것**: chunk_size (예: 300 / 500 / 1000, overlap은 비례해서 60/100/200)
- **고정**: 검색 방식은 실험①의 승자로

| chunk size | chunk 수 | hit@3 | 특이 실패 사례 |
|---|---|---|---|
| 300 | | | |
| 500 | | | |
| 1000 | | | |

- 관찰 포인트: 표가 깨진 부분은 큰 chunk가 유리한가? 고정 크기 vs 문단 기반도 비교하면 금상첨화
- (선택 심화) 관찰노트에서 제목 패턴(로마자 Ⅰ/숫자 1., 들여쓰기 0)을 발견했다면 **제목 기반 chunking**을 3번째 방식으로: 제목~다음 제목을 한 조각으로 묶고 "Ⅰ 사업개요 > 1. 관련 근거" 같은 제목 경로를 chunk에 메타데이터로 저장. 공고문처럼 구조가 뚜렷한 문서에서 강력하며, 검색 대상에 제목 경로를 포함시키면 hit@k가 오르는지도 실험거리

## 실험⑤ Hybrid (RRF) (Week 7~8)

- **가설 예시**: "두 방식의 실패 문항이 서로 다르므로, 순위 결합(RRF)이 단독보다 높을 것"
- 구현 (10줄, 직접):

```
RRF(문서) = 1/(60 + BM25 순위) + 1/(60 + 임베딩 순위)
→ 각 방식의 top-20 순위를 구해 위 점수로 재정렬 → top-k
(점수 크기가 달라도 '순위'만 쓰므로 안전. 60은 관례 상수)
```

| 임베딩 단독 | | |
| Hybrid (RRF) | | |

# Week 8 — 실패 케이스 10개 분류 (이 단계의 최종 결과물)

실험들의 `misses` 문항을 모아 **한 건씩 부검**합니다. 분류 기준:

| 분류 | 판정 방법 | 예시 |
|---|---|---|
| **검색 실패** | 정답 chunk가 top-k에 없음 (`is_hit` False) | 질문 "보조금" ↔ 문서 "지원금" 어휘 불일치 |
| **생성 실패** | 정답 chunk가 top-k에 **있는데** 답이 틀림/환각/불필요한 "정보 없음" | 3개 chunk 중 엉뚱한 것을 근거로 답변 |
| **데이터 문제** | `docs/text/`에서 정답 정보 자체가 깨져 있음 | 표 추출 시 "3년 이내" 줄 깨짐 |

부검 절차: 실패 문항 → ① top-k chunk를 직접 열어본다 → ② 정답이 있나? 없으면 원문 텍스트를 검색해본다 → ③ 원문에도 깨져 있으면 데이터 문제, 원문엔 있는데 검색이 못 찾으면 검색 실패, chunk에 있는데 답이 틀리면 생성 실패
