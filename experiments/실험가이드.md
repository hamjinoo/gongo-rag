# 검색 실험 가이드

이 문서는 “다음에 무엇을 바꾸고, 결과에서 무엇을 봐야 하는지”만 빠르게 확인하는 체크리스트입니다.

자세한 개념은 [`docs/EVALUATION.md`](../docs/EVALUATION.md), 현재 실제 결과는
[`retrieval-evaluation-dev.md`](./retrieval-evaluation-dev.md)에서 확인합니다.

## 현재 기준선

- 문서: `docs/text`의 TXT 3개
- 청킹: paragraph, 700자, overlap 120자
- 검색: BM25 / Chroma / RRF / Reranker
- 평가 질문: 전체 36개 중 개발용 normal 질문 20개
- 지표: Hit@1·3·5, MRR, nDCG@1·3·5, 평균·p95 지연 시간
- 현재 결론: 후보 7개가 Hit@1 0.85, 평균 약 4.20초로 현재 dev 기준 최선입니다.

## 실험할 때 지킬 규칙

1. 설정을 고르는 동안에는 `dev` 질문만 사용합니다.
2. 한 번에 한 가지만 바꿉니다.
3. 비교할 때 문서, 청크, 질문, `k`는 같게 유지합니다.
4. 정확도와 속도를 함께 기록합니다.
5. 잘 안 된 질문도 지우지 말고 원인을 적습니다.
6. 최종 설정을 고른 뒤에만 `test`를 한 번 실행합니다.

`test`를 자꾸 보면 시험 답을 외운 뒤 실력이 늘었다고 착각하는 것과 같습니다.

## 다음 실험 순서

### 1. Reranker 후보 수 줄이기 — 완료

후보 10·7·5개를 비교했습니다.

- 5개: 약 48.2% 빨라졌지만 정답 2개를 후보에서 잃어 폐기
- 7개: Hit@1 0.85, MRR 0.925를 기록하고 약 33.1% 빨라져 채택
- 자세한 결과: [`reranker-candidate-comparison-dev.md`](./reranker-candidate-comparison-dev.md)

```powershell
.venv\Scripts\python.exe src\run_retrieval_evaluation.py `
  --split dev `
  --systems reranker `
  --ks 1,3,5 `
  --rerank-candidates 5
```

후보 5개로 Hit@10을 요청하면 실제 구현은 결과 10개를 만들기 위해 후보도 10개로
늘려야 합니다. CLI는 이제 이런 모순된 설정을 거부합니다.

### 2. 더 작은 로컬 Reranker 비교하기

현재 모델보다 가벼운 다국어 모델을 후보 7개 조건으로 비교합니다.

- 기대: 응답 시간 감소
- 확인: 한국어 질문의 순위가 나빠지지 않는가?

### 3. Cohere Rerank 비교하기

API 기반 Reranker를 로컬 모델과 비교합니다.

- 비교 항목: 검색 품질, 응답 시간, 비용, 개인정보 전송 여부
- API 키나 비용이 필요하므로 별도 실험으로 남깁니다.

### 4. 설정 잠그고 test 한 번 실행하기

dev에서 모델을 선택한 다음 후보 수와 모델 설정을 고정합니다. 그 뒤에만 `test` 결과를
최종 포트폴리오 수치로 기록합니다.

## 실험 기록 템플릿

매번 별도 주간 기록을 만들 필요는 없습니다. 아래 항목만 PR 본문이나 새 실험
Markdown에 복사해서 사용합니다.

```text
가설:
바꾼 것:
그대로 둔 것:
실행 명령:
결과(정확도/속도):
실패 질문과 원인:
결정(채택/보류/폐기):
```

## 언제 청킹이나 토크나이저를 다시 바꾸나?

점수가 낮다는 이유만으로 모든 설정을 한꺼번에 바꾸지 않습니다. 실패 질문을
살펴봤을 때 아래 증거가 있을 때만 해당 단계를 다시 실험합니다.

- 정답 문장이 두 청크로 잘렸다면 청킹을 바꿉니다.
- 필요한 청크는 있으나 키워드 검색만 실패하면 한국어 토크나이저를 비교합니다.
- 후보에는 들어왔지만 1등이 아니면 Reranker를 개선합니다.
- 검색은 맞았는데 답변이 틀리면 생성과 Ragas 평가 단계에서 다룹니다.
