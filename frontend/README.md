# DocLens Trace

DocLens Trace는 **AI가 어떤 문장을 보고 답했는지 원문까지 따라가는 화면**입니다.

학교에서 친구가 “책에 그렇게 쓰여 있어”라고 말하면 우리는 “몇 쪽인데?”라고
물어볼 수 있습니다. 이 화면도 똑같습니다. 답변 옆의 근거를 누르면 실제 문서의
해당 페이지와 문장을 보여줍니다.

## 한눈에 보는 3개 화면

| 화면 | 누구를 위한 화면인가요? | 여기서 확인하는 것 |
|---|---|---|
| `/` | 질문하는 사람 | 답변, 답변에 실제로 쓰인 근거, 추가로 찾았지만 쓰지 않은 문서 |
| `/verify/1` | 답변을 검증하는 사람 | 주장과 원문의 연결, 파일명, 페이지, 정확한 원문 문장 |
| `/admin/trace/q_20260720_0412` | 개발자·관리자 | BM25·벡터·rerank 점수, 사용·제외 이유, 단계별 시간 |

일반 사용자 화면에는 검색 점수를 보여주지 않습니다. 점수는 답을 이해하는 데
필요하지 않고 오히려 혼란을 줄 수 있기 때문입니다. 자세한 숫자는 관리자 화면에
따로 모았습니다.

## 실행

Node.js 22.13 이상이 필요합니다.

```powershell
cd frontend
npm install
npm run dev
```

브라우저에서 `http://localhost:3000`을 열면 됩니다.

```powershell
npm run build
npm test
npm run lint
```

위 명령은 각각 배포용 빌드, 세 화면의 데이터·경로 검사, 코드 품질 검사를
실행합니다.

## 지금 실제로 되는 것

- 답변의 인용 칩이나 근거 카드를 누르면 알맞은 원문 화면으로 이동합니다.
- 근거 번호를 바꾸면 문서, 페이지, 강조 문장, 주장이 함께 바뀝니다.
- 검색 과정은 처음에는 접혀 있고 사용자가 원할 때만 펼칩니다.
- 관리자 화면에서 전체·사용됨·제외됨 필터를 바꿀 수 있습니다.
- 문서 추가 버튼은 PDF, Word, HWP, 이미지 파일 선택 창을 엽니다.
- 화면 크기가 작아지면 카드와 표가 모바일에 맞게 재배치됩니다.

현재 답변과 검색 기록은
[`app/lib/mock-trace.ts`](app/lib/mock-trace.ts)의 목업 데이터입니다.
즉, 프런트엔드의 구조와 이동은 완성됐지만 아직 Python RAG가 실시간으로 답을
보내지는 않습니다.

## 다음 연결 작업

기존 `src/rag_workflow.py`의 결과를 FastAPI로 감싸고 아래 두 API를 연결하면
됩니다.

1. `POST /api/documents`
   - PDF·DOCX·HWP·이미지를 보냅니다.
   - 추출과 chunking이 끝난 문서 ID, 파일명, 페이지 정보를 돌려받습니다.
2. `POST /api/query`
   - 질문과 검색할 문서 ID를 보냅니다.
   - 답변, 인용 근거, 관련 문서, 관리자 추적 기록을 돌려받습니다.

프런트엔드가 기대하는 핵심 응답 모양은 다음과 같습니다.

```json
{
  "query_id": "q_20260720_0412",
  "question": "1분기 매출 감소의 주요 원인은 무엇인가요?",
  "answer": "근거 번호가 포함된 최종 답변",
  "citations": [
    {
      "id": "1",
      "claim": "답변의 주장",
      "source_filename": "문서.pdf",
      "page": 14,
      "chunk_id": "ch_1q_014_02",
      "excerpt": "실제 원문 문장",
      "verified": true
    }
  ],
  "related_documents": [],
  "trace": {
    "durations": {},
    "chunks": []
  }
}
```

Python의 현재 응답에는 파일명, 페이지, chunk ID, 순위와 점수가 이미 있습니다.
다음 단계에서는 답변의 `[근거 N]`을 `citations[].id`와 연결하고, 원문 강조에
필요한 정확한 문자 구간을 추가하면 됩니다.

## 면접에서 설명할 기술 포인트

- **왜 사용자와 관리자 화면을 나눴나요?**  
  사용자는 답과 출처에 집중하고, 개발자는 검색 점수와 탈락 이유를 분석해야
  합니다. 목적이 달라 화면도 분리했습니다.
- **왜 인용 ID가 필요한가요?**  
  답변의 주장, 검색 chunk, 파일명, 페이지, 원문 문장을 하나의 번호로 연결하기
  위해서입니다.
- **왜 인용 개수를 코드에서 계산하나요?**  
  글자로 `3건`을 따로 적으면 데이터가 바뀌었을 때 숫자가 틀릴 수 있습니다.
  실제 인용 배열 길이를 표시하면 항상 일치합니다.
- **왜 검색 과정이 접혀 있나요?**  
  신뢰가 필요한 사람은 열어볼 수 있게 하면서도 평소에는 답변 읽기를 방해하지
  않기 위해서입니다.
- **지금 목업을 먼저 만든 이유는 무엇인가요?**  
  백엔드 연결 전에 필요한 데이터 모양과 사용자 동선을 확정하면 API를 만들 때
  빠뜨리는 값이 줄어듭니다.
