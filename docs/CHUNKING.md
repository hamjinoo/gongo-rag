# 2번 작업: 긴 글을 검색용 조각으로 나누기

## 이 문서 읽는 방법

- 처음 공부한다면 `한 줄로 설명하면`부터 `다음 작업`까지 읽습니다.
- 면접을 준비한다면 `면접 대비 기술 설명`부터 읽습니다.
- 직접 실행할 때는 맨 아래 `개발자용 실행 부록`을 봅니다.

## 한 줄로 설명하면

긴 문서를 질문과 비교하기 좋은 작은 글 묶음인 **chunk**로 나누는
작업입니다.

텍스트 추출이 “책의 글자를 컴퓨터 글자로 옮기는 일”이었다면, chunking은
그 책에 검색용 포스트잇을 붙이는 일입니다.

```text
긴 문서
→ 작은 글 묶음 여러 개
→ 질문과 가장 비슷한 묶음 찾기
```

## 왜 문서 전체를 그대로 검색하지 않나요?

500페이지짜리 책에서 “신청 자격은?”이라는 질문의 답을 찾는다고
생각해봅시다. 책 전체를 한 덩어리로 비교하면 신청 자격과 상관없는 내용이
너무 많이 섞입니다.

반대로 한 글자씩 너무 작게 나누면 뜻을 알 수 없습니다.

```text
너무 큼: 신청 자격 + 지원 금액 + 일정 + 문의처가 모두 섞임
적당함: 신청 자격과 설명이 함께 있음
너무 작음: "창업 3년" / "이내 기업"으로 정답이 잘림
```

따라서 질문에 답할 만큼 충분한 문맥은 남기고, 필요 없는 내용은 줄이는
크기를 찾아야 합니다.

## 이번에 만든 두 가지 방법

### 1. 고정 길이 방식

글자 수만 보고 같은 크기로 자릅니다.

```text
원문: 1,500자
chunk 크기: 500자
overlap: 100자

chunk 1: 0~499
chunk 2: 400~899
chunk 3: 800~1,299
chunk 4: 1,200~끝
```

장점은 단순하고 결과를 예상하기 쉽다는 것입니다. 단점은 문장이나 표를
중간에서 자를 수 있다는 것입니다.

고정 길이는 최종 정답이라기보다 다른 방법과 비교하는 **기준선**입니다.

### 2. 문단 우선 방식

최대 크기 안에서 다음 순서로 안전한 경계를 찾습니다.

```text
문단 사이 빈 줄
→ 줄바꿈
→ 문장 끝
→ 단어 사이 공백
→ 그래도 없으면 글자 수로 자르기
```

제목, 문단, 문장을 가능한 한 함께 보존하므로 현재 기본값으로
사용합니다.

## Chunk 크기와 overlap은 무엇인가요?

### Chunk 크기

한 조각에 넣을 최대 글자 수입니다. 현재 시작값은 `700자`입니다.

- 너무 크면 관련 없는 내용이 섞입니다.
- 너무 작으면 정답 문장이 여러 조각으로 갈라집니다.

### Overlap

앞 chunk의 마지막 내용을 다음 chunk에도 조금 넣는 것입니다. 현재
시작값은 `120자`입니다.

```text
chunk 1: ...신청 대상은 창업 3년 이내 기업입니다.
chunk 2: 창업 3년 이내 기업입니다. 접수 기간은...
```

경계에 걸린 문장이 잘리는 문제를 줄일 수 있습니다. 그러나 overlap이 너무
크면 같은 내용을 여러 번 저장하고 검색하게 됩니다.

문단 우선 방식에서는 단어 중간을 자르지 않기 위해 overlap 크기와 가까운
안전한 시작점을 찾습니다. 따라서 실제 겹치는 글자 수는 설정값보다 조금
작을 수 있습니다.

## 어떤 근거로 나누나요?

현재 기준은 다섯 가지입니다.

1. **페이지를 넘지 않습니다.** 답변의 원본 페이지를 정확히 보여주기 위해서입니다.
2. **문단과 문장을 먼저 지킵니다.** 뜻이 이어지는 내용을 함께 두기 위해서입니다.
3. **최대 700자를 넘지 않습니다.** 검색 결과에 불필요한 내용을 줄이기 위해서입니다.
4. **앞뒤를 약 120자 겹칩니다.** 경계에서 문장이 잘리는 문제를 줄이기 위해서입니다.
5. **출처 정보를 붙입니다.** 파일명, 페이지, 추출 방법을 잃지 않기 위해서입니다.

`700/120`은 영원한 정답이 아닙니다. 지금은 이해하고 비교하기 좋은
시작점입니다. BM25와 이후 embedding 검색에서 같은 질문으로 크기를
비교하게 됩니다.

## Chunk에는 글자만 들어 있나요?

아닙니다. 검색 결과가 어디에서 왔는지 알 수 있도록 metadata를 함께
저장합니다.

```json
{
  "id": "dfa8b421529b-p1-c0",
  "text": "2026년 글로벌화 사전검증 지원사업...",
  "metadata": {
    "source_filename": "지원사업 공고.pdf",
    "source_sha256": "dfa8b421...",
    "file_type": "pdf",
    "page_number": 1,
    "page_label": "페이지 1",
    "extraction_method": "native",
    "chunk_index": 0,
    "page_chunk_index": 0,
    "start_char": 0,
    "end_char": 694,
    "strategy": "paragraph"
  }
}
```

나중에 AI가 답할 때 `지원사업 공고.pdf 1페이지`처럼 근거를 표시할 수
있습니다.

## 화면에서 사용하는 방법

1. `1. 문서 넣기` 탭에서 파일을 올립니다.
2. `텍스트 추출`을 누릅니다.
3. 아래의 `2. 글자 나누기`로 이동합니다.
4. `문단 우선` 또는 `고정 길이`를 선택합니다.
5. 최대 크기와 overlap을 정합니다.
6. `Chunk 만들기`를 누릅니다.
7. 각 chunk의 글자와 metadata를 확인합니다.
8. 필요하면 `Chunk JSON 받기`를 누릅니다.

화면에는 다음 숫자가 표시됩니다.

| 숫자 | 의미 |
|---|---|
| Chunk 수 | 문서가 몇 조각으로 나뉘었는지 |
| 평균 크기 | 보통 한 조각이 몇 글자인지 |
| 최소 크기 | 가장 짧은 조각의 길이 |
| 최대 크기 | 가장 긴 조각의 길이 |

## 실제 공고문 결과

저장된 실제 공고문 하나로 추출부터 chunking까지 실행했습니다.

```text
문서: 2026년 글로벌화 사전검증 지원사업 참여기업 모집 공고
페이지: 5
추출 글자: 5,623자
추출 방법: 일반 PDF 텍스트
분할 방법: 문단 우선
최대 크기: 700자
overlap: 120자

결과 chunk: 11개
평균 크기: 약 554자
최소 크기: 118자
최대 크기: 694자
```

모든 chunk가 700자 이하였고 파일명, 페이지 번호, 추출 방법이
유지되는 것을 확인했습니다.

## 공부하면서 직접 확인할 것

같은 문서를 두 방식으로 각각 나눠보면 됩니다.

```text
실험 A: 고정 길이 700 / overlap 120
실험 B: 문단 우선 700 / overlap 120
```

그다음 다음 네 가지만 비교합니다.

1. 제목과 설명이 같은 chunk에 있나요?
2. 신청 자격, 날짜, 금액이 중간에서 잘리지 않았나요?
3. 표의 한 행이 여러 chunk로 흩어지지 않았나요?
4. 같은 문장이 너무 많이 반복되지 않았나요?

지금은 점수를 만들 필요가 없습니다. 두 방식의 실패 사례를 눈으로
이해하는 것이 이번 학습 목표입니다.

## 현재 한계

- 문단 구분이 없는 PDF는 글자 수 기준으로 잘릴 수 있습니다.
- 복잡한 표의 행과 열 구조를 이해하지는 않습니다.
- 페이지 끝의 제목과 다음 페이지의 설명을 한 chunk로 합치지 않습니다.
- 글자 수 기준이므로 embedding 모델의 실제 token 수와 다를 수 있습니다.
- 의미가 바뀌는 지점을 AI 모델로 판단하는 semantic chunking은 아직 하지 않습니다.
- 현재는 같은 chunk를 BM25 메모리 색인과 ChromaDB에 함께 저장해 검색합니다.

## 면접 대비 기술 설명

### 30초 설명

> 추출된 문서를 검색 단위로 변환하는 chunking 계층을 구현했습니다.
> 고정 길이 방식을 기준선으로 유지하고, 문단·줄·문장·단어 경계를 순서대로
> 찾는 문단 우선 방식을 기본값으로 추가했습니다. 페이지를 넘지 않도록
> 분할해 인용 위치를 보존하고, 각 chunk에 파일 해시, 페이지, 추출 방식,
> 원문 문자 위치를 metadata로 저장했습니다. 기존 BM25 인터페이스를
> 깨뜨리지 않으면서 새 구조화 문서 파이프라인을 추가했습니다.

새 파이프라인은 `chunk_document()`를 사용하고 `DocumentChunk` 목록을
반환합니다. 이렇게 하위 호환성을 지키면서 다음 RAG 구조에 필요한 metadata를
추가했습니다.

### DocumentChunk 모델

`DocumentChunk`는 다음 책임을 가집니다.

```text
검색할 본문
원본 문서 식별
페이지 인용
추출 품질 추적
문서 안 순서 복원
원문 문자 위치 확인
분할 전략 기록
```

ID는 기본적으로 다음 규칙을 사용합니다.

```text
{source_sha256 앞 12자}-p{페이지}-c{페이지 안 순서}
```

예:

```text
dfa8b421529b-p1-c0
```

파일명이 바뀌어도 내용이 같으면 같은 SHA-256을 갖기 때문에 문서 내용
기준으로 추적할 수 있습니다.

### 고정 길이 알고리즘

```python
start = 0
end = min(start + chunk_size, text_length)
next_start = end - overlap
```

마지막 chunk가 문서 끝에 도달하면 반복을 종료합니다. 이전 구현처럼 이미
끝까지 포함된 뒤 불필요한 작은 chunk를 하나 더 만들지 않습니다.

검증 조건은 다음과 같습니다.

- `chunk_size > 0`
- `overlap >= 0`
- `overlap < chunk_size`
- 모든 원문 글자가 최소 하나의 chunk에 포함
- 각 `start_char:end_char`가 실제 본문과 일치

### 문단 우선 알고리즘

현재 위치에서 `chunk_size`만큼의 최대 범위를 정하고, 그 범위의 뒤쪽
55% 구간부터 안전한 끝 지점을 찾습니다.

우선순위는 다음과 같습니다.

1. `\n\n`: 문단 경계
2. `\n`: 줄 경계
3. `.`, `?`, `!`: 문장 경계
4. 공백: 단어 경계
5. 경계가 없으면 최대 글자 수

앞부분에서 너무 일찍 자르지 않도록 뒤쪽 55%부터 검색합니다. 이 값 역시
휴리스틱이며 검색 평가 후 조정할 수 있습니다.

다음 chunk의 시작점도 overlap 목표 위치 근처에서 줄이나 단어 경계를
찾습니다. 이 때문에 문단 우선 방식의 실제 overlap은 설정값과 정확히
같지 않을 수 있습니다.

### 페이지를 넘지 않는 이유

각 `ExtractedPage`를 독립적으로 나눕니다.

장점:

- 답변에 정확한 페이지를 표시할 수 있습니다.
- OCR 페이지와 일반 페이지를 구분할 수 있습니다.
- 원문 문자 위치가 페이지 안에서 명확합니다.

단점:

- 페이지 마지막 제목과 다음 페이지 설명이 분리될 수 있습니다.
- 글자가 적은 페이지는 매우 짧은 chunk가 생길 수 있습니다.

현재는 인용 가능성을 우선했습니다. 이후 parent-child retrieval을 적용하면
페이지와 더 큰 문맥을 함께 사용할 수 있습니다.

### 중복 문서 처리

`chunk_documents()`는 기본적으로 `source_sha256`이 같은 문서를 한 번만
처리합니다. 같은 PDF를 두 번 올렸을 때 동일한 chunk가 검색 결과를
차지하는 것을 막기 위해서입니다.

중복을 유지해야 하는 실험에서는 `deduplicate=False`를 사용할 수 있습니다.
이때 ID에 `copy` 번호를 붙여 충돌을 막습니다.

### UI 상태 관리

추출된 문서 해시 목록을 `source_signature`로 저장합니다. 업로드 문서가
바뀌면 이전 문서의 chunk 결과를 보여주지 않습니다.

Chunk 결과와 적용 설정은 Streamlit `session_state`에 저장합니다. 사용자가
슬라이더를 바꾸기만 했을 때는 이전 결과임을 알 수 있도록 실제 적용된
설정을 따로 표시합니다.

### 테스트 전략

이번 단계에서 추가한 핵심 테스트는 다음과 같습니다.

- 파일명·SHA-256·페이지·추출 방법 metadata 보존
- 서로 다른 페이지의 텍스트가 한 chunk에 섞이지 않음
- `start_char:end_char`와 원문 일치
- 문단 경계를 고정 글자 수보다 우선
- 빈 페이지 제외
- 같은 내용의 중복 문서 제외
- 중복을 유지할 때 ID 충돌 방지
- 통계 계산
- 잘못된 크기와 overlap 거절
- Streamlit에서 추출 후 chunk 생성·미리보기

현재 자동 테스트는 기존 기능을 포함해 총 39개입니다.

실제 5페이지 공고문도 추출부터 분할까지 실행하여 크기 제한과 metadata를
확인했습니다.

### 기술 선택과 트레이드오프

| 선택 | 이유 | 한계 |
|---|---|---|
| 글자 수 기준 | 모델과 무관하고 한글에서도 설명·재현이 쉬움 | 실제 token 수와 다름 |
| 문단 우선 | 의미 단위 보존과 구현 투명성 | 문서 구조가 깨지면 효과 감소 |
| 700/120 기본값 | dev 평가를 재현하는 기준선 | 다른 크기와 비교해 고른 최적값은 아님 |
| 페이지별 분할 | 인용과 오류 추적이 쉬움 | 페이지 사이 문맥이 끊길 수 있음 |
| SHA-256 중복 확인 | 파일명과 무관하게 같은 내용 탐지 | 문서 일부만 같은 경우는 탐지 못함 |
| 직접 구현 | 경계 선택 이유를 이해하고 테스트 가능 | 고급 semantic chunking 기능 없음 |

### 예상 면접 질문

#### Q1. 왜 처음부터 semantic chunking을 사용하지 않았나요?

고정 길이와 문단 기반 방식은 동작을 설명하고 실패 원인을 확인하기 쉽습니다.
먼저 재현 가능한 기준선을 만든 뒤 retrieval 점수와 비용을 비교해야 semantic
chunking의 이점을 공정하게 판단할 수 있습니다.

#### Q2. 왜 700자와 overlap 120자를 선택했나요?

최종 최적값이 아니라 한글 공고문의 여러 문장을 함께 담으면서 화면에서
검수하기 좋은 시작값입니다. BM25와 embedding 검색을 연결한 뒤 크기별
Hit@k, chunk 수, 처리 시간을 비교해 결정할 계획입니다.

#### Q3. 왜 token이 아니라 글자 수로 나눴나요?

현재는 특정 embedding 모델에 종속되지 않는 입력 계층을 만드는 단계입니다.
한글 token 수는 tokenizer마다 달라집니다. 모델이 정해지면 최대 token 제한을
추가 검사하되, 재현 가능한 글자 수 기준도 함께 유지할 수 있습니다.

#### Q4. Overlap이 크면 항상 좋은가요?

경계의 문맥은 보존되지만 저장량, embedding 비용, 검색 결과 중복이
증가합니다. 너무 큰 overlap은 같은 근거가 상위 결과를 차지하게 만들 수
있으므로 retrieval 평가로 결정해야 합니다.

#### Q5. 왜 페이지를 넘지 않게 했나요?

정부 공고문 RAG에서는 답변의 파일명과 페이지를 보여주는 것이 중요합니다.
페이지를 섞으면 인용 위치가 모호해집니다. 다만 페이지 사이 문맥 단절이라는
단점이 있어 이후 parent-child 구조로 보완할 수 있습니다.

#### Q6. 표는 어떻게 처리하나요?

현재 추출 단계에서 표를 줄 단위 텍스트로 만들고, chunker는 줄 경계를
우선합니다. 복잡한 셀 병합과 헤더 관계는 보존하지 못합니다. 이후 표 전용
parser 또는 표 전체를 별도 child document로 저장하는 방식을 비교할 수
있습니다.

#### Q7. Chunk 품질은 어떻게 평가할 건가요?

이번 단계에서는 사람이 경계와 metadata를 검수합니다. 검색 단계에서는
동일한 질문으로 크기·전략별 Hit@1·3·5, MRR, 평균 후보 중복률과 처리 시간을
비교합니다.

#### Q8. 기존 BM25 코드를 왜 바로 새 모델로 바꾸지 않았나요?

한 번에 교체하면 기존 동작이 깨졌을 때 원인을 찾기 어렵습니다. 기존 dict
인터페이스를 유지하고 구조화된 `DocumentChunk`를 추가해 회귀 테스트를
통과시켰습니다. 현재는 별도 `BM25ChunkRetriever`가 새 모델을 받아 기존
점수 계산기에 연결합니다.

#### Q9. 같은 문서를 여러 번 올리면 어떻게 하나요?

파일 bytes의 SHA-256을 비교해 같은 내용이면 기본적으로 한 번만 chunking
합니다. Chroma 단계에서는 모델 설정, chunk 본문과 metadata의 해시로
collection을 나누고 저장된 ID·본문·metadata도 다시 확인합니다.

#### Q10. LangChain에는 어떻게 연결했나요?

각 `DocumentChunk`를 다음처럼 변환하면 됩니다.

```python
Document(
    id=chunk.id,
    page_content=chunk.text,
    metadata={**chunk.metadata, "chunk_id": chunk.id},
)
```

이 문서를 Chroma vector store에 전달했습니다. BM25도 같은 원본
`DocumentChunk`를 사용하므로 현재 구현된 RRF가 공통 chunk ID로 결과를 결합합니다.
