# 3번 작업: 질문과 같은 단어가 있는 조각 찾기

## 이 문서 읽는 방법

- 처음 공부한다면 `한 줄로 설명하면`부터 `다음 작업`까지 읽습니다.
- 면접을 준비한다면 `면접 대비 기술 설명`부터 읽습니다.
- 직접 실행할 때는 맨 아래 `개발자용 실행 부록`을 봅니다.

## 한 줄로 설명하면

문서를 잘라 만든 여러 chunk 중에서 **질문의 중요한 단어가 들어 있는
chunk를 점수순으로 찾는 작업**입니다.

```text
문서에서 글자 꺼내기
→ 긴 글을 chunk로 나누기
→ 질문과 같은 단어가 있는 chunk 찾기 ← 지금 여기
→ 비슷한 뜻의 chunk도 찾기
→ 두 검색 결과 합치기
```

도서관에 “청년 창업 지원 금액”이 적힌 부분을 찾아달라고 했다고
생각해봅시다. BM25는 각 포스트잇에 `청년`, `창업`, `지원`, `금액`이
있는지 확인하고, 더 중요한 단어가 잘 들어 있는 포스트잇을 먼저
가져옵니다.

## 이번 작업에서 얻어야 하는 것

코드 공식을 전부 외우는 것이 목표는 아닙니다. 다음 네 가지를 설명할 수
있으면 됩니다.

1. BM25는 같은 **단어**를 기준으로 찾는 검색기입니다.
2. 한국어는 조사가 붙으므로 검색 전에 단어를 잘 나눠야 합니다.
3. 검색 결과에는 점수뿐 아니라 파일명과 페이지도 남아야 합니다.
4. BM25가 못 찾는 “같은 뜻, 다른 단어”는 다음 embedding 검색이 맡습니다.

## BM25는 어떻게 순서를 정하나요?

BM25는 크게 세 가지를 봅니다.

### 1. 질문 단어가 문서에 있는가?

질문이 `지원 금액`이면 두 단어가 있는 chunk가 점수를 얻습니다. 아무
단어도 없으면 이 프로젝트에서는 검색 결과에서 제외합니다.

### 2. 드문 단어인가?

모든 chunk에 있는 `사업`보다 몇 개의 chunk에만 있는 `보증금`이 더
구체적인 단서일 가능성이 큽니다. 그래서 드문 단어에 더 큰 점수를 줍니다.

### 3. 단어가 알맞게 반복되고 있는가?

질문 단어가 한 번 있는 것보다 여러 번 있는 문서가 보통 더 관련 있습니다.
다만 `지원`을 100번 썼다고 점수를 100배 주지는 않습니다. 긴 문서가 무조건
유리해지지 않도록 문서 길이도 보정합니다.

```text
같은 단어가 있음       → 점수 증가
드문 단어가 있음       → 더 크게 증가
같은 단어가 반복됨     → 증가하지만 점점 증가 폭이 작아짐
문서가 불필요하게 김   → 길이 보정
```

## 한국어에서는 왜 Kiwi가 필요한가요?

컴퓨터가 공백으로만 단어를 나누면 다음 두 표현을 다른 단어로 볼 수
있습니다.

```text
질문: 신청 자격이 무엇인가요?
문서: 신청 자격은 창업 3년 이내 기업입니다.

공백 분리: 자격이 ≠ 자격은
Kiwi 분석: 자격 = 자격
```

`자격이`의 `이`, `자격은`의 `은`은 조사입니다. Kiwi 형태소 분석기는
조사와 어미를 떼고 검색에 중요한 명사, 동사, 형용사, 숫자 등을
남깁니다.

현재 화면에서는 두 방법을 비교할 수 있습니다.

| 방법 | 하는 일 | 용도 |
|---|---|---|
| Kiwi 한국어 형태소 | 조사·어미 분리, 동사·형용사 기본형 사용 | 현재 추천 검색 |
| 기본 단어 분리 | 문장부호 제거 후 한글·영문·숫자 덩어리 분리 | 비교용 기준선 |

Kiwi가 모든 한국어를 완벽하게 이해한다는 뜻은 아닙니다. 어느 품사를
남길지에 따라 결과가 달라지므로 나중에 고정 질문셋으로 비교해야 합니다.

## 화면에서 사용하는 방법

1. `문서 실험실` 탭에서 PDF, DOCX, 이미지 또는 TXT를 올립니다.
2. `텍스트 추출`을 누릅니다.
3. `2. 글자 나누기`에서 `Chunk 만들기`를 누릅니다.
4. `3. BM25 키워드 검색`으로 내려갑니다.
5. 처음에는 `Kiwi 한국어 형태소 (추천)`을 선택합니다.
6. `신청 자격은 어떻게 되나요?`처럼 질문합니다.
7. `BM25 검색`을 누릅니다.
8. 점수가 높은 순서와 파일명·페이지·일치 단어를 확인합니다.
9. 같은 질문을 `기본 단어 분리`로도 검색해 차이를 봅니다.

검색 결과의 점수는 확률이 아닙니다. `3.2점`이 “정답일 확률 32%”라는
뜻이 아니라, **현재 문서들 안에서 서로 순서를 비교하기 위한 값**입니다.

## 검색 결과에서 무엇을 봐야 하나요?

```text
1위 · score 4.217 · 지원사업 공고.pdf · 페이지 2
일치 단어 · 신청, 자격, 창업
```

다음 네 가지를 확인합니다.

1. 질문의 답이 정말 1~3위 chunk 안에 있나요?
2. 질문에서 어떤 단어를 검색에 사용했나요?
3. 파일명과 페이지가 원본과 맞나요?
4. 단어는 같은데 뜻이 다른 잘못된 결과가 위에 있지는 않나요?

결과를 JSON으로 받아 이후 실험 기록이나 다른 검색기와의 비교에도 사용할
수 있습니다.

## 실제 공고문으로 확인한 결과

저장소에 있는 실제 5페이지 공고문으로 추출부터 검색까지 실행했습니다.

```text
문서: 2026년 글로벌화 사전검증 지원사업 참여기업 모집 공고
추출 방법: 일반 PDF 텍스트
분할 방법: 문단 우선 700자 / overlap 120자
생성 chunk: 11개
검색 질문: 지원 대상은 누구인가요?
Kiwi 검색 단어: 지원 / 대상 / 누구
검색 결과: 점수가 0보다 큰 상위 3개
상위 결과 페이지: 2, 3, 2
```

모든 결과에 원본 파일명, 페이지 번호, 추출 방법이 유지되는 것을
확인했습니다.

## 공부하면서 직접 해볼 작은 실험

다음 세 질문을 같은 문서에서 검색합니다.

```text
A. 신청 자격은 무엇인가요?
B. 신청 자격이 무엇인가요?
C. 누가 신청할 수 있나요?
```

먼저 기본 단어 분리와 Kiwi를 비교합니다.

- A와 B는 조사만 다르므로 Kiwi에서 더 비슷한 결과가 나와야 합니다.
- C는 같은 뜻이지만 단어가 다릅니다. BM25가 놓칠 수 있습니다.
- C를 놓치는 것은 버그라고 단정하지 않습니다. 다음 embedding 검색을
  추가해야 하는 이유를 보여주는 사례입니다.

메모에는 다음만 적으면 됩니다.

```text
질문:
원하는 근거가 나온 순위:
기본 분리와 Kiwi의 차이:
이상한 결과:
내가 이해한 이유:
```

## 현재 한계

- 질문과 문서가 같은 뜻이어도 단어가 다르면 놓칠 수 있습니다.
- 오타, 띄어쓰기 오류, OCR 오류에 영향을 받습니다.
- 표의 행과 열 관계를 따로 이해하지는 않습니다.
- 모든 chunk를 메모리에서 직접 채점하므로 문서가 매우 많으면 느려집니다.
- BM25 점수만으로 답이 맞는지 판단할 수는 없습니다.
- 이 BM25 검색 자체는 embedding을 사용하지 않으며, Chroma 결과와 RRF로 합치지는 않습니다.

### BM25 공식

현재 구현은 BM25Okapi에서 널리 사용하는 다음 식을 사용합니다.

```text
IDF(t) = ln((N - df(t) + 0.5) / (df(t) + 0.5) + 1)

score(D, Q)
  = 질문의 각 단어 t에 대해
    IDF(t) × f(t,D) × (k1 + 1)
    ─────────────────────────────
    f(t,D) + k1 × (1 - b + b × |D| / avgdl)
    를 모두 더한 값
```

| 기호 | 의미 |
|---|---|
| `N` | 전체 chunk 수 |
| `df(t)` | 단어 `t`가 등장한 chunk 수 |
| `f(t,D)` | chunk `D` 안에서 `t`가 나온 횟수 |
| `|D|` | 해당 chunk의 token 수 |
| `avgdl` | 평균 chunk token 수 |
| `k1=1.5` | 단어 반복 횟수의 포화 정도 |
| `b=0.75` | 문서 길이 보정 강도 |

`k1`과 `b`는 현재 비교 가능한 시작값이며 최종 최적값은 아닙니다.

### 색인 단계의 성능 처리

초기화할 때 다음 값을 한 번 계산합니다.

- 각 chunk의 token 목록
- 각 chunk의 token 수
- 각 chunk의 `Counter` 기반 단어 빈도
- 평균 길이
- 단어별 document frequency

질문마다 `list.count()`로 같은 chunk를 반복 탐색하지 않고 미리 만든
`Counter`에서 빈도를 조회합니다. 현재는 모든 chunk의 점수를 계산한 뒤
정렬하므로 구현은 투명하지만 대규모 corpus에는 역색인이나 전문 검색
엔진이 더 적합합니다.

### 한국어 tokenizer 설계

`KiwiTokenizer`는 다음 종류를 검색어로 유지합니다.

```text
명사 NN*
동사 VV*
형용사 VA*
보조 용언 VX*
대명사 NP
수사 NR
관형사 MM
부사 MAG, MAJ
어근 XR
외국어 SL
숫자 SN
한자 SH
```

조사와 어미는 제외하고, 분석 결과에 lemma가 있으면 기본형을 사용합니다.
예를 들어 활용된 동사를 같은 기본형으로 비교할 가능성이 커집니다.

품사 선택은 정답이 아니라 실험 변수입니다. 너무 많이 남기면 잡음이
늘고, 명사만 남기면 동작이나 상태를 나타내는 중요한 표현을 잃을 수
있습니다.

Kiwi 객체 생성 비용을 반복하지 않도록 tokenizer factory에 작은
`lru_cache`를 사용합니다. Streamlit의 chunk별 retriever도
`cache_resource`로 재사용합니다.

### 기존 코드와 호환한 방법

기존 `BM25` 클래스와 공백 분리 `tokenize()` 함수는 손계산 학습 테스트가
같은 값을 내도록 유지했습니다.

새 파이프라인은 별도의 `BM25ChunkRetriever`가 담당합니다.

```python
retriever = BM25ChunkRetriever(chunks, tokenizer_name="kiwi")
results = retriever.search("신청 자격이 무엇인가요?", k=5)
```

이 검색기는 기존 점수 계산기를 합성해서 사용하고, 검색 결과에
`DocumentChunk`를 다시 연결합니다. 기존 기준선을 없애지 않고 한국어
문서 파이프라인을 확장한 구조입니다.

### SearchResult와 출처 추적

각 결과는 다음 값을 가집니다.

```text
rank
score
matched_terms
chunk.text
chunk.metadata
```

점수와 본문만 반환하면 나중에 답변의 출처를 복원하기 어렵습니다. 처음부터
원본 `DocumentChunk`를 유지하므로 파일명, 해시, 페이지, OCR 여부, 원문
위치와 분할 전략을 답변 단계까지 전달할 수 있습니다.

### 점수가 0인 결과를 제외한 이유

기본 BM25는 모든 문서에 점수를 매길 수 있고, 질문 단어가 하나도 없으면
0점입니다. 단순히 `top-k`를 자르면 관련 없는 0점 chunk도 결과처럼 보일 수
있습니다.

현재 검색기는 양수 점수만 반환합니다. 이것은 “양수면 정답”이라는 뜻이
아니라, 최소한 질문 token 하나는 일치했다는 조건입니다. 최종 답변 거절
기준은 검색 평가와 reranker 점수를 보고 별도로 정해야 합니다.

### UI 상태와 캐시

chunk 내용·ID·문자 위치·분할 전략으로 signature를 만들고 결과 상태에
저장합니다. 다음 중 하나가 바뀌면 이전 결과를 숨기고 다시 검색하도록
안내합니다.

- 원본 chunk
- 질문
- tokenizer
- top-k

이렇게 해야 새 설정 아래에 이전 검색 결과가 남아 있는 것처럼 보이는
UI 오류를 막을 수 있습니다.

### 테스트 전략

이번 단계에서 확인한 내용은 다음과 같습니다.

- 기존 BM25 손계산 IDF·순위 유지
- 빈 corpus와 잘못된 top-k 처리
- 문장부호를 제거하는 기본 tokenizer
- Kiwi가 조사 제거와 lemma를 사용하는지
- `자격이` 질문이 `자격은` 문서의 `자격`과 연결되는지
- 일치 결과가 먼저 오는지
- 무관한 질문은 빈 결과를 반환하는지
- 파일명·페이지·추출 방법 metadata 보존
- 기존 Hit@k 평가 함수에 연결 가능한 text adapter
- 업로드 → 추출 → chunking → 검색 Streamlit 흐름

기존 기능을 포함한 자동 테스트 **48개가 모두 통과**했습니다. 실제
5페이지 PDF도 11개 chunk로 만든 뒤 Kiwi BM25로 검색해 상위 결과와
metadata를 확인했습니다.

### 기술 선택과 트레이드오프

| 선택 | 이유 | 한계 |
|---|---|---|
| 직접 구현한 BM25 | 공식과 실패 이유를 설명하고 손계산 검증 가능 | 대규모 색인 최적화 부족 |
| Kiwi 기본값 | 한국어 조사·어미 문제 완화 | 모델 로딩 비용과 품사 선택 필요 |
| 기본 분리 유지 | tokenizer 개선 효과를 비교할 기준선 | 한국어 활용 처리 약함 |
| 양수 결과만 반환 | 무관한 0점 결과 노출 방지 | 양수라고 정답인 것은 아님 |
| metadata 포함 결과 | 페이지 인용과 오류 추적 가능 | 객체와 캐시 키가 더 커짐 |
| 메모리 색인 | 데모가 단순하고 재현 가능 | 앱 재시작 시 다시 색인 |

### 예상 면접 질문

#### Q1. Embedding 검색이 있는데 왜 BM25가 필요한가요?

고유명사, 사업명, 날짜, 금액, 공고 번호처럼 정확한 문자열이 중요한
질문에서 BM25가 강합니다. 의미 검색과 실패 유형이 다르므로 두 결과를
RRF로 합치면 서로의 약점을 보완할 수 있습니다.

#### Q2. 왜 한국어에 공백 분리만 사용하지 않았나요?

한국어는 명사 뒤에 조사가 붙고 용언이 활용됩니다. `자격이`와 `자격은`을
공백 분리하면 다른 token이지만 형태소 분석 후에는 공통 핵심어 `자격`을
얻을 수 있습니다.

#### Q3. 왜 명사만 남기지 않았나요?

명사는 정밀도가 높을 수 있지만 `제외하다`, `가능하다`, `마감되다` 같은
동작과 상태를 잃을 수 있습니다. 현재는 명사·동사·형용사 등을 시작값으로
두고 고정 질문셋에서 품사 조합별 Hit@k를 비교할 계획입니다.

#### Q4. `k1=1.5`, `b=0.75`는 어떻게 정했나요?

일반적인 시작값이며 이 corpus에서 최적이라고 주장하지 않습니다. chunk
전략과 tokenizer를 먼저 고정한 뒤 dev 질문셋에서 소수 후보를 비교하고,
test 질문셋은 최종 확인에만 사용해야 합니다.

#### Q5. BM25 점수를 threshold로 바로 사용할 수 있나요?

corpus 크기와 질문 token의 희귀도에 따라 점수 범위가 달라져 절대 확률처럼
사용할 수 없습니다. 질문 유형별 분포와 정답 유무를 평가한 뒤 거절 기준을
정하거나 reranker·답변 검증과 함께 사용해야 합니다.

#### Q6. 왜 라이브러리 BM25를 바로 쓰지 않았나요?

현재 단계의 목적은 공식을 이해하고 회귀 테스트 가능한 기준선을 만드는
것입니다. 운영 규모가 커지면 Elasticsearch/OpenSearch 같은 역색인 기반
검색기로 교체하더라도 현재 구현을 품질 비교 기준으로 남길 수 있습니다.

#### Q7. 시간 복잡도와 확장성은 어떤가요?

현재 질문마다 모든 chunk를 채점하고 정렬하므로 대략 `O(N log N)`입니다.
포트폴리오 데모에는 충분하지만 대규모 환경에서는 역색인으로 후보를
줄이고 heap 기반 top-k 또는 검색 엔진을 사용해야 합니다.

#### Q8. 검색 결과의 출처는 어떻게 보존하나요?

색인 순서와 `DocumentChunk` 목록을 함께 유지합니다. 점수가 나온 index를
원본 chunk로 되돌려 파일명, 페이지, 추출 방식, 문자 위치를 포함한
`SearchResult`를 만듭니다.

#### Q9. BM25가 실패하면 바로 embedding을 추가하면 되나요?

먼저 실패가 tokenization, chunk 경계, OCR, 동의어 중 무엇 때문인지
기록합니다. 같은 뜻인데 단어가 달라 실패한 사례가 확인되면 embedding
검색을 추가할 근거가 생깁니다.

#### Q10. RRF에는 어떻게 연결했나요?

BM25와 Chroma가 각각 chunk ID 순위를 반환하게 합니다. 점수 크기는 서로
비교하지 않고 각 검색기의 순위만 사용해 다음처럼 합칩니다.

```text
RRF 점수 = Σ 1 / (상수 k + 각 검색기의 순위)
```

같은 chunk ID로 결과를 합치고 원본 metadata는 그대로 유지합니다. 그
후 상위 후보에만 CrossEncoder reranker를 적용합니다.

### LangChain과 연결할 위치

다음 단계에서 각 chunk를 LangChain 문서로 바꿉니다.

```python
from langchain_core.documents import Document

documents = [
    Document(
        page_content=chunk.text,
        metadata=chunk.metadata,
    )
    for chunk in chunks
]
```

이 문서를 embedding한 뒤 ChromaDB에 저장합니다. BM25는 현재
`DocumentChunk`를 계속 사용하거나 동일 ID를 통해 LangChain Document와
연결할 수 있습니다.
